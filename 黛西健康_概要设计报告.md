## 2. 系统概述

### 2.1 建设目标

系统面向智慧养老和健康服务场景，为管理员、老人用户和上门服务人员提供统一入口，实现：

- 老人档案、健康数据、设备、报告、用药和资产的集中管理。
- 商品服务、订单、工单、售后和评价的统一业务链路。
- 服务人员审核、派单、查询和工单状态处理。
- 活动、资讯、健康讲堂等运营内容在管理端和用户端同步。
- 基于 JWT 和角色权限的统一身份认证与访问控制。
- 仪表盘和业务分析页面的数据汇总展示。

### 2.2 系统边界

当前项目是一个前后端分离的单体应用：

- 一个 Vue 3 前端同时承载管理端、用户端和服务人员端。
- 一个 Spring Boot 后端提供统一的 `/api/v1/**` REST API。
- 一个 MySQL 数据库保存三端共享的业务数据。
- 上传文件保存在后端本地 `uploads/` 目录。
- Redis 依赖和配置已保留，但当前核心业务不依赖 Redis。

当前没有拆分微服务，也没有接入云对象存储、消息队列、第三方支付、短信或真实硬件平台。

### 2.3 用户角色

| 身份 | `roleType` | 主要入口 | 权限范围 |
|---|---|---|---|
| 超级管理员/后台员工 | `staff` | `/dashboard` 等后台路由 | 按 `role.permissions` 控制后台模块及操作 |
| 老人用户 | `elderly` | `/portal/user` | 仅访问本人资料、健康、资产、活动和工单数据 |
| 服务人员 | `service` | `/portal/service` | 仅访问本人资料和分配给本人的工单 |

后台角色数据位于 `role` 表。超级管理员使用通配权限，其他员工按模块授予查看或编辑权限。

## 4. 前端设计

### 4.1 统一登录与身份分流

三类身份共用 `/login`。登录成功后，前端根据后端返回的 `roleType` 跳转：

```text
staff    → 管理端第一个有查看权限的模块
elderly → /portal/user
service → /portal/service
```

路由守卫负责：

- 未登录访问业务路由时返回登录页。
- 有 Token 但缺少用户资料时调用 `/api/v1/auth/profile` 恢复会话。
- 老人和服务人员不能进入后台管理路由。
- 后台员工无模块查看权限时跳转到第一个可访问模块。
- 本地缓存异常时清理无效登录状态。

### 4.2 管理端页面

管理端使用 `AdminLayout.vue` 作为主布局，核心页面包括：

| 路由 | 页面 | 主要功能 |
|---|---|---|
| `/dashboard` | 工作台 | 核心指标、业务概览 |
| `/schedule` | 预约看板 | 预约/工单新增、编辑和状态展示 |
| `/users` | 用户列表 | 查询、新增、编辑、删除用户 |
| `/users/:id` | 用户详情 | 完整资料、健康数据、设备、用药、报告、工单等 |
| `/user-assets/:resource` | 用户资产 | 优惠券、积分、等级、积分记录等 |
| `/user-health/:resource` | 健康扩展 | 健康设置、设备、报告、用药、健康数据 |
| `/service/*` | 服务管理 | 服务人员、审核、工单及人员/用户查询 |
| `/products` | 商品服务管理 | 商品与服务统一目录 |
| `/operations/:resource` | 运营管理 | 活动、报名、资讯、讲堂、食谱等 |
| `/trade/*` | 交易管理 | 订单、售后、评价 |
| `/analytics` | 数据分析 | 用户、交易、工单、评价等图表 |
| `/system/*` | 系统管理 | 员工、角色、操作日志 |

多数后台资源使用 `GenericListView.vue`，通过资源配置复用表格、筛选、表单和 CRUD 操作。复杂用户资料由 `UserDetailView.vue` 独立实现。

### 4.3 老人用户端

老人用户端集中在 `UserPortalView.vue`，当前功能如下：

| 功能 | 可查看 | 可修改 |
|---|---:|---:|
| 完整个人资料 | 是 | 是 |
| 头像 | 是 | 是，可选预置头像或上传 |
| 健康数据 | 是，含体重和心率折线图 | 否 |
| 用药记录 | 是 | 否 |
| 绑定设备 | 是 | 可修改本人设备信息 |
| 健康报告 | 是 | 否 |
| 优惠券和积分 | 是 | 否 |
| 商品服务 | 是 | 否 |
| 工单 | 是，仅本人 | 可选择服务人员并创建 |
| 社区活动 | 是，分“我的活动”和“可选择活动” | 只能报名，不能创建活动 |
| 健康资讯 | 是 | 否 |
| 健康讲堂 | 是 | 否 |

用户端和管理端直接读取同一套业务表，不建立用户端数据副本。管理端更新用户资料、活动、资讯或讲堂数据后，用户端重新请求即可读取最新数据。

### 4.4 服务人员端

服务人员端集中在 `ServicePortalView.vue`，当前已实现：

- 查看本人姓名、手机号、头像、服务类型、区域、入职时间、审核状态和评分。
- 查看仅分配给本人的工单列表。
- 查看本人工单详情。
- 更新本人工单状态：待服务、服务中、已完成、已取消。
- 完成工单时记录完成时间。

当前尚未实现服务人员自行编辑完整资料、上传服务过程附件、位置签到、排班日历、收入统计、评价明细和消息通知。

### 4.5 头像与上传交互

`AvatarPicker.vue` 为老人用户和超级管理员复用的头像组件：

- 支持 6 个系统预置头像。
- 支持 JPG、JPEG、PNG、WebP。
- 前端先校验格式和原图大小，再执行图像处理和上传。
- 上传后保存后端返回的 `/uploads/**` 相对地址。
- 展示时由统一资源地址函数转换为可访问 URL。

| `ElderlyPortalController` | `/api/v1/elderly`，老人用户端全部接口 |
| `ServiceAppController` | `/api/v1/service-app`，服务人员端接口 |
| `ServiceController` | 服务人员、审核和管理端工单 |
| `ProductController` | 商品服务目录 |
| `TradeController` | 订单、售后和评价 |
| `OperationController` | 活动、资讯、讲堂等运营资源 |
| `PhaseResourceController` | 健康、资产、报名等通用扩展资源 |

| `SystemController` | 员工、角色、协议和日志 |
| `UploadController` | 本地文件上传 |

### 5.2 服务层

`AdminDataService` 和 `PortalDataService` 是主要服务接口：

- `JdbcAdminDataService`：管理端查询、资源 CRUD、账号同步、工单和运营管理。
- `JdbcPortalDataService`：老人及服务人员的本人数据查询和受限写操作。
- `MockDataService`、`MockPortalDataService`：Mock Profile 的演示实现。
- `ActivityEnrollmentSql`：集中活动真实人数、容量校验和兼容计数字段同步。

当前 JDBC SQL 主要集中在服务实现中。`UserMapper` 和 `UserEntity` 保留了 MyBatis 入口，但项目不是完整的“所有表均 Mapper 化”结构。

### 5.3 统一响应

所有正常业务响应使用：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

主要错误码：

| 错误码 | 含义 |
|---:|---|
| 1001 | 参数错误 |
| 1002 | Token 缺失、无效或过期 |
| 1003 | 无权限 |
| 1005 | 业务校验失败，如账号密码错误、名额已满、上传不合法 |
| 5000 | 未分类服务器异常 |

分页数据使用 `PageResult`，返回总数和列表。

### 5.4 事务设计

类级服务默认使用 Spring `@Transactional`。关键写流程增加业务约束：

- 创建工单前校验商品、用户和服务人员，避免产生半成品订单或工单。
- 活动报名使用 `READ_COMMITTED`，先锁定活动行，再统计真实报名人数并校验名额。
- 管理端新增有效报名、恢复取消报名、迁移活动或更换报名用户时执行同样的活动锁和容量校验。
- 用户只能更新本人资料、本人设备和本人业务数据。
- 服务人员工单查询和状态更新始终带当前服务人员 ID 条件。

## 6. 核心业务设计

### 6.1 登录和资料同步

```text
手机号/密码
   ↓
查询 account + 对应 profile
   ↓
BCrypt 校验（旧明文由启动迁移器转换）
   ↓
签发 JWT，返回账号、roleType、权限
   ↓
前端按身份进入对应工作台
```

系统处于统一账号迁移阶段：

- `account` 保存通用登录字段。
- `admin_profile`、`elderly_profile`、`service_profile` 保存身份关联资料。
- `staff`、`user`、`service_personnel` 仍是当前主要业务资料来源。
- `data.sql` 在启动时幂等创建或同步统一账号资料。
- 管理员头像默认值仅在 `staff.avatar_url` 为空时写入，不能覆盖已经保存的头像；随后再同步到 `account.avatar_url`。

### 6.2 用户资料和健康数据

管理端可以查看并修改用户完整档案。用户端读取同一用户资料，并允许修改本人可维护字段。

健康数据按独立记录保存：

```text
health_data
├─ user_id
├─ data_type：weight、heart_rate 等
├─ value + unit
├─ record_date + record_time
└─ source + device_id
```

设计约束：

- 不把同一天的体重和心率合并为一条记录。
- 修改心率不会覆盖体重，修改体重也不会覆盖心率。
- 管理端和用户端进入页面时即根据已有记录生成趋势图。
- 用户端健康数据只读，不提供修改入口。
- 前端 `healthChart.js` 将管理端聚合数据和用户端原始数据转换为统一图表格式。

### 6.3 工单创建与派单

管理端和老人用户端创建工单时都必须明确选择服务人员：

```text
选择商品服务
  + 选择客户（用户端固定为当前用户）
  + 选择服务人员
  + 选择服务时间
           ↓
后端校验商品、客户、人员状态和审核状态
           ↓
按商品目录价格创建订单/工单
           ↓
三端读取同一 work_order
```

可选服务人员必须满足：

```text
service_personnel.status = 1
audit_status = '已通过'
```

管理端工单列表可按服务人员和用户查询。用户只能查看本人工单，服务人员只能查看分配给自己的工单。

### 6.4 活动与真实报名人数

管理端负责创建和维护活动，老人用户只能查看和报名。

用户端活动分组：

- 我的活动：当前用户最新报名状态为已报名或已到场。
- 可选择的活动：已发布、未结束、名额未满且当前用户未有效报名。
- 活动详情：封面、标题、状态、开始/结束时间、地点、名额、介绍、报名状态和报名时间。

报名人数不使用可能失真的 `activity.enrolled` 作为展示依据，而是实时统计 `activity_enroll`：

```text
按 activity_id + user_id 分组
          ↓
每名用户只取 id 最大的最新记录
          ↓
status = enrolled 或 attended → 计入
status = cancelled              → 不计入
          ↓
得到活动真实报名人数
```

用户端和管理端使用同一统计 SQL。`activity.enrolled` 仅作为兼容字段，在用户报名以及管理端新增、修改、删除报名后按真实人数回写。

为避免并发超额，用户端和管理端的有效报名写入均在 `READ_COMMITTED` 事务中锁定目标活动并校验真实人数。

### 6.5 运营内容同步

| 用户端内容 | 管理端数据源 | 用户端规则 |
|---|---|---|
| 社区活动 | `activity`、`activity_enroll` | 可查看和报名，不可创建 |
| 健康资讯 | `article` | 读取已发布内容 |
| 健康讲堂 | `video` | 读取已发布内容 |

三端不复制运营数据。管理端修改原记录后，用户端下一次请求读取同一数据库中的最新内容。

## 7. 接口概要设计

### 7.2 老人用户端接口

## 9. 安全与权限设计

### 9.1 身份认证

- 登录使用手机号和密码。
- 密码使用 BCrypt 验证。
- `PasswordMigrationRunner` 在启动时迁移旧明文密码。
- 登录成功签发 JWT，默认有效期 8 小时。
- JWT 包含账号身份信息，后端不接受前端伪造当前用户 ID。

### 9.2 后端权限

`JwtAuthFilter` 拦截 `/api/v1/**`：

1. 校验 Bearer Token。
2. 解析 `AuthenticatedUser` 并写入请求属性。
3. 调用 `PermissionService` 按身份、HTTP 方法和路径鉴权。
4. 无认证返回 1002，无权限返回 1003。

老人和服务人员的数据隔离在 SQL 层再次实现，不能只依赖前端路由隐藏。

### 9.3 前端权限

- 后台菜单按模块查看权限显示。
- 路由守卫禁止进入无权页面。
- 操作按钮可按编辑权限控制。
- 老人和服务人员固定进入自己的门户路由。

### 9.4 上传安全

- 后端根据上传分类限制扩展名和大小。
- 文件名进行清理并添加分类和时间戳。
- 上传目录通过 `/uploads/**` 映射为静态资源。
- `uploads/` 不纳入 Git。

当前上传存储为本地磁盘，生产部署应配置持久化目录、备份策略和访问域名。

## 12. 当前限制与后续建议

### 12.1 当前限制

- 后端服务层集中承载较多 SQL 和资源映射，模块继续增长时维护成本会上升。
- 数据库同时保留旧业务表和统一账号 Profile 表，仍处于兼容迁移结构。
- Redis 尚未用于会话、缓存或分布式锁。
- 文件存储为本地磁盘，不具备天然多实例共享能力。
- 服务人员端仅完成资料查看和工单基本处理。
- 用户端以单个综合页面承载较多功能，后续功能增长时需拆分。
- 当前活动人数使用实时聚合，数据量显著增大后需增加组合索引并评估性能。
- 尚未接入真实支付、短信、推送、硬件设备和第三方服务。

### 12.2 建议演进方向

1. 将超长 JDBC 服务按账号、用户健康、工单、活动和运营领域拆分。
2. 为 `activity_enroll(activity_id, user_id, id)`、工单人员和用户查询字段补充索引。
3. 完成统一账号 Profile 迁移后，明确旧表与新表的唯一主数据来源。
4. 为活动报名和工单创建增加真实 MySQL 集成测试与并发测试。
5. 将上传目录外置，增加文件病毒扫描、鉴权下载和生命周期管理。
6. 完善服务人员资料维护、服务过程记录、签到、排班和统计。
7. 拆分用户端页面并增加移动端适配和无障碍设计。
8. 增加配置中心、日志采集、监控告警、数据库备份和恢复演练。

## 13. 设计结论

当前版本已经形成一个可运行的智慧养老三端系统：统一前端负责身份分流，一个 Spring Boot 后端提供受 JWT 和 RBAC 保护的 API，一个 MySQL 数据库作为管理端、老人用户端和服务人员端的共享事实来源。

系统已完成用户健康管理、设备和报告查看、商品服务、显式派单、工单查询、活动报名、健康资讯、健康讲堂、后台运营、交易和系统管理等核心链路。健康数据和活动数据均在管理端与用户端共享；用户端健康数据只读；活动人数以最新有效报名记录实时统计；管理员头像在重启后保持持久化。

后续工作的重点不再是补齐基础页面，而是深化服务人员端、收敛账号迁移结构、拆分服务层、补充集成与并发测试，并完善生产部署能力。
